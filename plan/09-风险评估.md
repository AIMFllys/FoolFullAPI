# 09 — 风险评估与应对

---

## 风险矩阵

| 风险 | 概率 | 影响 | 等级 | 应对措施 |
|------|------|------|------|---------|
| Kimi/DeepSeek API 不可用 | 中 | 高 | 🔴 高 | 兜底回复替代 + 自动降级 |
| AI 被用户带偏，泄露整蛊意图 | 中 | 高 | 🔴 高 | 双重约束 + 输出过滤 + 兜底回复 |
| Kimi 在 Step 5 输出完整答案 | 中 | 高 | 🔴 高 | max_tokens 限制 + 后端截断 + 强制追加中断提示 |
| 大量用户涌入导致过载 | 低 | 高 | 🟡 中 | 速率限制 + 全局日限 + 扩容 |
| API 成本超出预算 | 低 | 中 | 🟡 中 | 全局日限 1000 次 + 用户 18 次上限 |
| 用户投诉/负面反馈 | 中 | 低 | 🟢 低 | 服务条款明确 + 友好彩蛋 |
| 数据清理任务未执行 | 低 | 高 | 🟡 中 | 手动备份 + 监控告警 |
| 域名/SSL 问题 | 低 | 高 | 🟡 中 | 提前配置 + 备用方案 |

## 应急预案

### 预案 A：AI API 完全不可用

```
触发条件：连续 3 次 API 调用失败（Kimi 或 DeepSeek）
应对措施：
1. 自动切换到"纯兜底模式"
2. 所有 6 步使用预设的兜底回复
3. 在回复前添加 2~4 秒随机延迟模拟思考
4. 记录日志，通知管理员
5. 每 5 分钟尝试恢复 API 调用
```

### 预案 B：Kimi API 不可用但 DeepSeek 可用

```
触发条件：Kimi 调用失败但 DeepSeek 正常
应对措施：
1. Step 1 和 Step 5 降级使用 DeepSeek
2. 降低 Step 1 的输出质量预期（分析可能不够深入）
3. Step 5 使用更严格的提示词约束 DeepSeek 输出
4. 记录日志，监控降级后的输出质量
```

### 预案 C：Step 5 Kimi 输出完整答案

```
触发条件：Kimi 在 Step 5 输出了完整的、可用的答案（没有中断）
应对措施：
1. 后端检测回复中是否包含 "⚠️ [系统提示]" 标记
2. 如果没有，强制在回复末尾追加中断提示
3. 如果回复超过 300 字，截断到 200 字处并追加中断提示
4. 截断位置选择在代码行中间或句子中间，模拟"被打断"
```

### 预案 D：服务器过载

```
触发条件：CPU > 90% 或内存 > 90%
应对措施：
1. 降低全局日限（1000 → 500）
2. 增加速率限制严格程度
3. 临时扩容服务器（如有条件）
```

### 预案 E：数据清理失败

```
触发条件：4月2日 00:05 检查数据仍存在
应对措施：
1. 手动执行清理脚本
2. 直接删除数据库文件并重建空表
3. 确认后重启服务
```

## Step 5 输出质量保障（重点）

Step 5 是整个流程的高潮，必须确保输出质量：

### 问题场景与应对

| 场景 | 检测方式 | 应对 |
|------|---------|------|
| Kimi 输出完整答案 | 检查是否包含中断标记 | 强制截断 + 追加标记 |
| Kimi 输出太短（<50字） | 检查长度 | 使用兜底回复 |
| Kimi 输出与问题无关 | 无法自动检测 | 接受（概率极低） |
| Kimi 泄露整蛊意图 | 关键词过滤 | 使用兜底回复 |
| Kimi 拒绝执行 | 检查是否包含"无法"/"抱歉" | 使用兜底回复 |

### 后端 Step 5 后处理逻辑

```typescript
function postProcessStep5(reply: string): string {
  // 1. 检查是否包含中断标记
  if (!reply.includes('⚠️') && !reply.includes('[系统提示]')) {
    // 没有中断标记，强制截断
    const cutPoint = Math.min(reply.length, 200);
    // 找到一个自然的截断点（代码行中间、句子中间）
    const truncated = reply.slice(0, cutPoint);
    return truncated + '\n\n⚠️ [系统提示] 检测到当前会话为愚人节特别版本，内容生成已暂停。';
  }
  return reply;
}
```
